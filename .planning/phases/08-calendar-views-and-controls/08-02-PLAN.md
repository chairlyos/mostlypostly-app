  <action>
Replace the stub GET /week handler with a full implementation. The endpoint returns an HTML fragment (not a full page) that gets inserted into the calendar-view-body element by the client-side switchView function from Plan 01. The fragment uses server-side safe() escaping on all user content (same pattern as the existing day panel fragment).

**Query params:**
- `req.query.week` accepts an ISO date string YYYY-MM-DD representing the start of the week (always a Sunday). Default: current week's Sunday.

**Server logic:**
1. Get salon timezone from DB: `SELECT timezone, facebook_page_token, instagram_business_id, tiktok_enabled, google_location_id FROM salons WHERE slug = ?`
2. Parse req.query.week as a Luxon DateTime in salon timezone. If invalid or missing, compute current week's Sunday: `DateTime.now().setZone(tz)`, then adjust to Sunday (if weekday is not 7, subtract weekday days; Luxon weekday 7 = Sunday).
3. Compute week range: weekStart (Sunday) to weekEnd = weekStart.plus({ days: 6 }).endOf('day').
4. Convert to UTC for DB query.
5. Query posts (same query as month view with same columns including vendor_campaign_id + vc.vendor_name JOIN).
6. Group posts by local salon date (same byDate Map pattern as month view).

**HTML fragment structure:**

Navigation row (inside the fragment so it updates on fetch):
- Previous week button with `data-week-nav="${prevWeekISO}"` class `week-nav-btn`
- Week label text: `${weekStart.toFormat('MMM d')} - ${weekEnd.toFormat('MMM d, yyyy')}` (e.g., "Mar 23 - Mar 29, 2026")
- Next week button with `data-week-nav="${nextWeekISO}"` class `week-nav-btn`
- prevWeekISO = weekStart.minus({ weeks: 1 }).toFormat('yyyy-LL-dd')
- nextWeekISO = weekStart.plus({ weeks: 1 }).toFormat('yyyy-LL-dd')
- Use same arrow SVG icons as month navigation

Day-of-week headers: grid grid-cols-7 with each column showing "Mon 3/24" format (3-letter day name + month/day).

Grid: single row, grid grid-cols-7, with each cell being a calendar-day-cell div at min-h-[200px] (taller than month cells). No day number inside the cell (shown in header instead). Show ALL posts per day (no 3-post limit, no "+N more").

Post cards: Use the exact same mini-card HTML as the month view including .calendar-post-card class, color bar via calendarCardBarClass(p), type label via calendarPillLabel(p), platform icons via platformIcons(salon, 'sm'), data-post-type, data-status, card-field-stylist, card-field-platforms, card-field-time, card-field-caption classes, data-draggable on manager_approved posts. Caption preview truncated to 60 chars per D-25.

Normalize post type for data-post-type attribute: vendor_campaign_id truthy maps to "vendor", before_after_post maps to "before_after", promotions maps to "promotion", celebration_story maps to "celebration", otherwise use post_type as-is.

Highlight today's cell with ring-2 ring-mpAccent ring-inset (same as month view).

The fragment includes a script block at the end for week-specific behavior:
- Week navigation: click handlers on .week-nav-btn elements that fetch the new week fragment and replace calendar-view-body content. After replacing, call window.applyFilters() and window.applyCardSettings() if those functions exist.
- SortableJS initialization on each .calendar-day-cell: same config as month view (group calendar-posts, pull/put true, draggable only data-draggable=true elements, animation 0, ghostClass opacity-40, onEnd calls POST /manager/calendar/reschedule with postId and newDate).
- Day cell click: call window.openDayPanel(cell.dataset.date) on click (skip if click was on a .calendar-post-card element).

**Also update the main page script (Plan 01 IIFE):**
- Expose applyFilters and applyCardSettings to window scope by adding `window.applyFilters = applyFilters;` and `window.applyCardSettings = applyCardSettings;` after their definitions. openDayPanel is already on window from Phase 7.
- Update switchView function so when view is 'week', the fetch URL includes the week query param if present in the current URL search params.
