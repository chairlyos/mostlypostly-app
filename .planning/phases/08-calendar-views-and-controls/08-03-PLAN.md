Replace the stub GET /agenda handler with a full implementation. The endpoint returns an HTML fragment (not a full page) inserted into the calendar-view-body element by switchView(). All user content is escaped via the existing safe() helper (same pattern as the day panel fragment).

**Server logic:**
1. Get salon from DB: `SELECT timezone, facebook_page_token, instagram_business_id, tiktok_enabled, google_location_id FROM salons WHERE slug = ?`
2. Compute 30-day range in salon timezone: rangeStart = today at start of day, rangeEnd = today + 30 days at end of day.
3. Convert to UTC for DB query.
4. Query posts — same query as month/week views with same columns (id, post_type, status, scheduled_for, published_at, stylist_name, image_url, image_urls, final_caption, base_caption, vendor_campaign_id, salon_post_number, vc.vendor_name). Only include future-facing posts: `status NOT IN ('draft', 'cancelled')` and `(scheduled_for BETWEEN ? AND ? OR published_at BETWEEN ? AND ?)`.
5. Group by local salon date using the same byDate Map pattern.
6. Sort dates chronologically.

**HTML fragment structure:**
No navigation arrows (per D-06). The main page JS already hides nav-arrows when agenda is active.

For each date that has posts, render a date-header row followed by post cards:
```html
<div class="mb-6">
  <h3 class="text-sm font-bold text-mpCharcoal mb-2 sticky top-0 bg-[#F8FAFC] py-1 z-10">${dateLabel}</h3>
  <!-- post cards -->
</div>
```
Where dateLabel is formatted as "Thursday, March 26" using `dt.toFormat('EEEE, MMMM d')`.

Post cards use the **same style as the day panel cards** (larger cards with thumbnail, full info). Each card:
- Has class `.agenda-post-card` (not `.calendar-post-card`) so filters can target both via the shared data attributes
- Has `data-post-type` and `data-status` attributes (same normalization as month/week views)
- Shows: thumbnail image (via toProxyUrl), stylist/vendor name, time display, platform icons (md size), type label pill, status badge, caption preview (120 chars, not 60)
- No action buttons in agenda view (those live in the day panel; clicking a card could open the day panel for that date)

Card HTML structure (matches day panel card style):
```html
<div class="agenda-post-card bg-white rounded-lg border border-mpBorder p-3 shadow-sm mb-2 cursor-pointer hover:border-mpAccent/40 transition-colors" data-post-type="${normalizedType}" data-status="${p.status}" data-date="${dateStr}">
  <div class="flex gap-3">
    ${thumbnail}
    <div class="flex-1 min-w-0">
      <div class="flex items-center justify-between mb-0.5">
        <span class="font-medium text-sm text-mpCharcoal truncate card-field-stylist">${nameDisplay}</span>
        <span class="text-xs text-mpMuted ml-2 flex-shrink-0 card-field-time">${timeDisplay}</span>
      </div>
      <div class="card-field-platforms">${platformIcons(salon, 'md')}</div>
      <div class="flex items-center gap-1.5 mt-1 flex-wrap">
        ${typeLabel}
        ${statusBadge(p.status)}
      </div>
      <p class="text-sm text-mpMuted line-clamp-2 mt-2 card-field-caption">${safe(preview)}</p>
    </div>
  </div>
</div>
```

Where thumbnail is either an img tag with the proxied image URL or a camera emoji placeholder div (same as day panel).

Add a click handler on agenda cards that opens the day panel for that date:
```javascript
<script>
(function() {
  document.querySelectorAll('.agenda-post-card').forEach(function(card) {
    card.addEventListener('click', function() {
      if (typeof window.openDayPanel === 'function') window.openDayPanel(card.dataset.date);

</script>
```

Empty state: If no posts in the 30-day window, show:

```html
  <p class="text-[11px] text-mpMuted mt-1">Post will be scheduled for this date. Time will be auto-assigned by the scheduler.</p>
</div>
` : ''}
```

If no `?date=` param, the form looks exactly as before (no date field shown).

**Changes to POST handler (around line 1164):**

3. After reading `stylist_id` and `caption_note` from req.body, also read:
```javascript
const scheduledDate = req.body.scheduled_date || '';
```

4. After the post is created and saved (the section where it redirects to /manager with a notice), if `scheduledDate` is a valid YYYY-MM-DD string, update the post's `scheduled_for` to that date at 10:00 AM in the salon's timezone (a reasonable default time):
```javascript
if (scheduledDate && /^\d{4}-\d{2}-\d{2}$/.test(scheduledDate)) {
  // Import DateTime from luxon at top of file if not already imported
  const { DateTime } = await import('luxon');
  const salon = db.prepare("SELECT timezone FROM salons WHERE slug = ?").get(salon_id);
  const tz = salon?.timezone || 'America/Indiana/Indianapolis';
  const localDt = DateTime.fromISO(scheduledDate + 'T10:00:00', { zone: tz });
  if (localDt.isValid) {
    const utcStr = localDt.toUTC().toFormat('yyyy-LL-dd HH:mm:ss');
    db.prepare("UPDATE posts SET scheduled_for = ? WHERE id = ?").run(utcStr, postId);
  }
}
```
Note: The post is already created with `manager_pending` status at this point. The scheduled_for date is advisory; the actual scheduling happens when the manager approves the post. But setting it ensures the calendar shows the post on the intended day.

Check if luxon is already imported at the top of manager.js. If not, add `import { DateTime } from "luxon";` at the top of the file.

Important: The `postId` variable must be accessible at this point in the handler. Check the existing code flow to identify where the new post ID is set. It may be stored as `post.id` or returned from `savePost()`. Read the POST handler carefully to find the correct variable name.
