# Quick Task 260321-1c1: add stylist-provided caption passthrough with heuristic detection and AI regenerate option - Context

**Gathered:** 2026-03-21
**Status:** Ready for planning

<domain>
## Task Boundary

When a stylist texts a photo + message body, the app currently always runs OpenAI Vision to generate a caption and discards the stylist's text (treating it as a label/direction). This feature adds a passthrough path: if the stylist's message looks like a real caption, keep it verbatim. On the preview page, show a prominent "Generate AI Version" button so they can still get AI output on demand.

</domain>

<decisions>
## Implementation Decisions

### Detection method
- **Pure heuristics only** — no AI classification call, zero extra API cost, runs in <1ms
- Heuristic signals that indicate a REAL caption (keep it):
  1. **8+ words** in the message body
  2. **Any emoji present** in the message body (strong signal regardless of word count)
  3. Starts with a capital letter
  4. Contains punctuation (period, !, ?, comma)
- A message passes if it meets (8+ words OR contains emoji) AND at least one of the other signals
- Short labels/tags like "Balayage ✂️" (1-2 words + emoji) should NOT be kept — the emoji-only rule applies when combined with meaningful length. Clarification: emoji alone is not enough; must also have 5+ words when emoji is the primary trigger.
- Implementation function: `isRealCaption(text)` in `messageRouter.js` or a new helper

### Portal UX (stylist preview page)
- Caption is shown in the edit textarea as-is (labeled or visually distinct — e.g. "Your caption" note above)
- A prominent blue **"Generate AI Version"** button is visible on the portal page
- Clicking it swaps the textarea content with AI-generated output (client-side fetch or same REDO path)
- The kept caption is pre-filled in `base_caption` field in DB — consistent with current draft flow

### SMS feedback
- When caption is KEPT: `"Your caption was kept! Review and approve here: {url}"`
- When caption triggers AI generation (short/tag-style): current flow unchanged — `"Here's your post preview: {url}"`

### Word count threshold
- Primary rule: **8+ words** to qualify via word count path
- **Emoji flag**: if message contains an emoji AND has 5+ words, treat as real caption
- This catches concise-but-proper captions while excluding bare tags like "Highlights ✨"

### Coordinator flow
- Same heuristic logic applies to coordinator-submitted posts — no branching

### Claude's Discretion
- Where exactly the "Generate AI Version" button lives on the stylist portal page (above/below textarea)
- Whether the portal shows a subtle "Kept your caption" badge vs just the button
- Exact wording of portal label ("Your caption" vs "Caption (kept from your message)")

</decisions>

<specifics>
## Specific Ideas

- The emoji check should use a Unicode regex or a simple `.match(/\p{Emoji}/u)` test
- `isRealCaption(text)` should be a pure function (easy to unit test) — extracted to a helper or inline in messageRouter.js
- The AI regenerate on the portal page can reuse the existing REDO/REGENERATE endpoint or call a new `/stylist/regenerate-caption` route that accepts `post_id` and returns new caption JSON
- The `base_caption` field should store the stylist-provided text so the manager edit form (after today's fix) shows their clean caption without metadata

</specifics>
