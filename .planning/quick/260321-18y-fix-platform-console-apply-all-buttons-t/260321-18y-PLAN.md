In `src/routes/vendorAdmin.js` around line 1008, the inline `onchange` handler on the Apply All checkboxes uses an arrow function:

```
onchange="document.querySelectorAll('.col-${plat}').forEach(cb => { cb.checked = this.checked; })"
```

The arrow function captures `this` from the outer scope (window), not the checkbox element. `this.checked` is always `undefined`/falsy, so it only unchecks.

Fix: Replace the arrow function body to use `event.target.checked` instead of `this.checked`. Change line 1008 to:

```
onchange="var v=this.checked; document.querySelectorAll('.col-${plat}').forEach(function(cb){ cb.checked=v; })"
```

Key points:
- Use `var v=this.checked` BEFORE the forEach (in the inline handler context, `this` IS the element — it is only broken inside the arrow function callback)
- Actually, `this` in inline `onchange` IS the element. The bug is that inside the `.forEach(cb => ...)` arrow callback, `this` still correctly refers to the element in an inline handler context. Let me re-examine...

Actually, in an inline event handler attribute, `this` refers to the element. Arrow functions preserve the `this` of the enclosing scope. In an inline handler, the enclosing scope's `this` IS the element. So `this.checked` inside the arrow SHOULD work.

Re-examine the actual bug: the checkboxes have class `col-${plat}` but the template literal `${plat}` is evaluated server-side in the Node.js template string. So `col-facebook`, `col-instagram`, `col-gmb`, `col-tiktok` are the actual class names on the body checkboxes (line 1037). The onchange selector `.col-${plat}` also evaluates server-side to `.col-facebook` etc. This should be correct.

The REAL issue may be that the inline `onchange` attribute itself is being stripped or not firing due to CSP or the earlier CSP fix (task 260320-ate) converting some handlers to event delegation but missing the routing table section. Check whether the `onchange` attribute actually exists in rendered HTML.

Alternative root cause: the Apply All checkbox is inside `<thead>` and the column checkboxes are inside `<tbody>`. The `querySelectorAll('.col-${plat}')` searches the entire document so this should not matter.

Another possibility: the `onchange` on the Apply All toggle fires and sets checkboxes correctly, but the Tailwind CSS toggle visual state (`peer-checked`) does not update because programmatically setting `.checked` does not trigger the CSS `:checked` pseudo-class refresh in all browsers. Actually, `.checked = true` DOES update `:checked` in CSS.

Given uncertainty about the exact cause, take a DEFENSIVE approach — replace the inline `onchange` with event delegation in the existing `<script>` block at the bottom of the page. This is more robust and consistent with the CSP fix pattern from 260320-ate:

1. Remove the `onchange="..."` attribute from line 1008 entirely
2. Add a `data-apply-all` attribute with the platform name: `data-apply-all="${plat}"`
3. In the `<script>` section (around line 1529+), add event delegation:

```javascript
document.addEventListener('change', function(e) {
  var toggle = e.target.closest('[data-apply-all]');
  if (!toggle) return;
  var plat = toggle.getAttribute('data-apply-all');
  var checked = toggle.checked;
  document.querySelectorAll('.col-' + plat).forEach(function(cb) {
    cb.checked = checked;
  });
});
```

This approach:
- Removes inline handler (consistent with 260320-ate CSP fix pattern)
- Uses `e.target.closest()` delegation (same pattern as other handlers in the file)
- Uses regular function (no `this` ambiguity)
- Reads `.checked` from the actual toggle element
